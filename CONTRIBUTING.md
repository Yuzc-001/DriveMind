# Contributing

Thanks for helping improve DriveMind.

## What good contributions look like
DriveMind is a product about calm reliability, clear boundaries, and human collaboration. Good contributions usually make the project:
- easier to trust
- easier to install
- easier to understand
- easier to maintain.

## Areas to contribute
- `skill/`: behavior, references, and templates
- `docs/`: product explanation, examples, and operating guidance
- `scripts/`: installation and packaging improvements
- `assets/logo/`: careful visual refinements that stay within the current design system
- `.github/workflows/`: validation and release automation
- `README.md` and `README.zh-CN.md`: public entry points that should stay aligned.

## Before you open a change
- keep changes small and reviewable
- explain the problem, not only the fix
- update docs when behavior or install paths change
- update examples if the user-facing workflow changes
- add a changelog entry for meaningful project-facing changes
- keep `README.md` and `README.zh-CN.md` in sync when public-facing messaging changes
- follow the tone in `docs/brand-kit.md`.

## Local checks
Before opening a pull request, make a reasonable effort to:
- parse the PowerShell scripts without syntax errors
- run shell syntax checks for the bash installers
- confirm local markdown links still resolve
- read both README entry points once from top to bottom for consistency.

## Licensing and brand
By contributing, you agree that your contribution is licensed under the repository license in `LICENSE.md`.
Contributing code or docs does not grant permission to reuse the DriveMind name or logo outside the terms described in `TRADEMARKS.md`.

## Collaboration expectations
- be respectful and direct
- favor clarity over cleverness
- surface uncertainty early
- avoid shipping behavior that blurs safety boundaries without discussion.
